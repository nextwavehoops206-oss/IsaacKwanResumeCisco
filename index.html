Lab #01

Palo Alto Firewall Factory Reset

PAN-OSPA-220U-BootMaintenance ModeWeb GUI
// Purpose

The purpose of this lab is to learn how to factory reset a Palo Alto firewall so we can access it later in the year for other projects.

// Background

In this lab, we practiced doing a factory reset on a Palo Alto Networks firewall — wiping the device back to its default settings. This is something network admins often need to do if a firewall is repurposed, given to a new team, or if troubleshooting requires a clean slate.

Palo Alto firewalls run PAN-OS, and during boot-up they use the U-Boot bootloader, which gives options for normal startup or entering maintenance mode. The maintenance partition is where factory resets, software reloads, and recovery tasks are performed.

Knowing how to reset a firewall is a foundational skill — it ensures you can always return a device to a clean state and prepare it for new configurations.

// Steps
  1. During the boot process, when prompted, press any key to stop the autoboot countdown.
  2. At the bootloader prompt, type maint and press Enter to boot into the maintenance partition.
  3. In the maintenance menu, select the Factory Reset option and confirm.
  4. Wait for the reset to complete. The firewall will reboot to factory defaults.
  5. Connect your PC to the management port of the firewall using an Ethernet cable.
  6. Open a browser and navigate to the default management IP address.
  7. Log in using the default credentials (username: admin, password: admin) to access the web GUI and confirm the reset was successful.
// Problems Encountered

We encountered a minor problem when trying to log into the firewall after the reset. We performed the reset a second time to restart the process cleanly, which resolved the login issue.

// Conclusion

By completing this lab, we gained practical experience performing a factory reset on a Palo Alto firewall and accessing its management interface. We learned how to interrupt the normal boot sequence, enter the maintenance partition, and restore the device to its default state — essential skills for any firewall administrator.

Lab #02

Fortinet SOHO Firewall Configuration

FortiGateNAT / SNATDHCPUTM / AntivirusNGFW
// Purpose

The purpose of this lab is to create a Small Office/Home Office (SOHO) network that allows private traffic from an internal network to reach the public internet, while implementing NAT and security features to protect internal devices.

// Background

In a SOHO architecture, the Fortinet firewall acts as the primary security gateway — bridging the untrusted public internet and the private internal network. Because internal devices use private IP addresses that cannot be routed on the internet, FortiGate uses Source NAT (SNAT) to mask the internal infrastructure behind a single public-facing WAN IP.

This configuration also leverages Next-Generation Firewall (NGFW) capabilities, moving security from the network layer up to the application layer. By applying Unified Threat Management (UTM) profiles, the firewall performs stateful inspection and deep packet analysis on all outbound traffic — enabling policies like web filtering to block URLs and antivirus scanning to catch threats in transit.

This multi-layered defense protects the SOHO network in ways traditional firewalls simply cannot.

// Steps
  1. Connect an Ethernet cable from the WAN port on the firewall to an Ethernet switch. Connect a second cable from port E0/1 to your PC.
  2. Open a browser and go to 192.168.1.99 (the Fortinet default management IP). Log in with username admin.
  3. Go to System → Settings and configure a hostname and the correct time zone.
  4. Go to Network → Interfaces to configure LAN, WAN, and NAT settings.
  5. Select the LAN interface and set its Alias to Office LAN. Change the IP/Netmask from the default 192.168.1.99/24 to 192.168.10.1/24.
  6. Under the LAN interface DHCP settings, configure the address range for leasing to internal clients.
  7. Select the WAN interface and configure its Alias and interface members as needed. Press OK to save.
  8. Go to Policy and Objects → Firewall Policy and press Create to add a new policy.
  9. Configure a LAN-to-WAN policy that allows all sources to all destinations, with NAT enabled. Press OK to save.
// Problems Encountered

We had trouble finding the correct default IP for the Fortinet web GUI. Our previous firewall used 192.168.1.1, but the Fortinet's default management IP is 192.168.1.99.

We were also given a firewall that was already pre-configured by a previous group. We guessed the credentials, got locked out, and ultimately had to factory reset it before starting our own configuration.

// Conclusion

This lab taught me how to configure a SOHO environment from scratch using a Fortinet FortiGate. I now understand how NAT enables private networks to reach the internet, how UTM profiles add application-layer security, and how Fortinet integrates its features into a unified platform — skills directly applicable to real-world network administration.

Lab #03

VPN Configuration — Palo Alto PA-220

GlobalProtectPA-220SSL/TLSNATRemote Desktop
// Purpose

The purpose of this lab is to configure a VPN on the PA-220 firewall that allows secure remote access to a network from any location. This enables both encrypted connectivity and remote desktop access.

// Background

A firewall acts as a barrier between a trusted internal network and an untrusted external network, allowing or blocking traffic based on security rules. The Palo Alto PA-220 is an enterprise-grade appliance designed for branch offices, small offices, and remote deployments.

For this lab, all configurations were made through the PA-220's web-based GUI — a centralized, graphical interface that is becoming the standard for managing network devices as the industry moves away from CLI-only management.

The lab built on an existing SOHO configuration. We created certificates to define VPN policies, configured DHCP so IP addresses are assigned dynamically (eliminating manual IP configuration errors), and set up NAT to allow multiple internal devices to share a single public IP.

The end goal was a working GlobalProtect VPN tunnel through which we could remote desktop into a PC on the internal network from an outside device.

// Steps
  1. Configure initial network interfaces and zones in the PA-220 web GUI.
  2. Navigate to Network → GlobalProtect → Portals and create a new portal to handle VPN client connections.
  3. Under Network → GlobalProtect → Gateways, configure the GlobalProtect gateway settings.
  4. Create a certificate for the VPN portal and gateway using the Certificate Management section.
  5. Configure tunnel interfaces and assign them to a VPN security zone.
  6. Enable IP address pools so the firewall can assign IPs to connecting VPN clients.
  7. Configure NAT policies to allow VPN tunnel traffic to reach the internal network.
  8. Create outbound security policy rules permitting traffic from the VPN zone to the internal zone.
  9. Allow users to connect without a valid portal certificate (for testing purposes).
  10. On the client device, open Remote Desktop Connection and enter the target machine's IP to verify full VPN connectivity.
// Problems Encountered

We had problems with our certificate configuration and IP addressing in multiple stages of the lab. Some instructions were unclear, which led to significant errors early on. We had to retrace our steps carefully to find and fix each issue — this was the most challenging and time-consuming part of the lab.

// Conclusion

This lab provided valuable hands-on experience building a VPN on a Palo Alto firewall end-to-end. I now understand how GlobalProtect works, how certificates secure VPN authentication, and how NAT and security policies enable encrypted remote access. I would be able to replicate this configuration for personal or corporate use.

Lab #04

Wireless Access Point — Fortinet + FreeRADIUS

FortiGateWPA2 EnterpriseFreeRADIUSUbuntu ServerVirtualBox
// Purpose

The purpose of this lab is to configure secure wireless connectivity through a Fortinet firewall using both WPA2 Personal (passphrase-based) and WPA2 Enterprise (RADIUS-authenticated) security modes.

// Background

Fortinet is a cybersecurity company founded in Sunnyvale, California. Unlike vendors that sell separate products for each security function, Fortinet focuses on deep integration — combining firewall, wireless LAN, AI-powered threat detection (FortiGuard), and cloud security into a unified platform.

Fortinet's next-generation firewalls support built-in wireless controller functionality, letting administrators manage access points directly from the firewall. They also integrate with major cloud providers including AWS, Azure, and Google Cloud via the FortiCNAPP platform.

WPA2 Enterprise requires a RADIUS server for authentication, unlike WPA2 Personal which only needs a shared passphrase. FreeRADIUS — which authenticates roughly a third of all internet users — was used in this lab to back the enterprise SSID. It was deployed on an Ubuntu Server virtual machine running in Oracle VirtualBox.

// Steps
  1. In FortiGate, go to WiFi & Switch Controller → Managed FortiAPs.
  2. Authorize the connected access point: select it → Authorization → Authorize. Verify the connection status.
  3. Go to WiFi & Switch Controller → SSIDs → Create New → SSID to create the first SSID.
  4. Configure a WPA2 Personal SSID: set an IP/Netmask, enable DHCP, name the SSID, and set the security mode to WPA2 Personal with a passphrase.
  5. Download Ubuntu Server 24.04 LTS from ubuntu.com/download/server and Oracle VirtualBox from virtualbox.org.
  6. In VirtualBox, create a new VM, select the Ubuntu ISO image, customize the hostname and credentials, then go to Settings → Expert → Network → Bridged Adapter.
  7. Boot the VM and log in. Run sudo apt update to refresh package lists.
  8. Install FreeRADIUS: sudo apt install freeradius -y.
  9. Configure the RADIUS clients file: sudo nano /etc/freeradius/3.0/clients.conf. Add client details at the bottom, then save with CTRL+O → Enter → CTRL+X.
  10. Configure a test user: sudo nano /etc/freeradius/3.0/users. Add testuser Cleartext-Password := "password123" at the bottom. Save and exit.
  11. Back in FortiGate, create a WPA2 Enterprise SSID under WiFi & Switch Controller → SSIDs, and integrate the RADIUS server.
  12. Edit your FortiAP profile (WiFi & Switch Controller → FortiAP Profiles) to activate both SSIDs.
  13. Go to Policy & Objects → Firewall Policy → Create New Policy and add a policy allowing wireless traffic to reach the internet.
  14. Test from an external device: connect using the WPA2 Personal passphrase or the WPA2 Enterprise credentials from FreeRADIUS.
// Problems Encountered

We ran into issues integrating the virtual machine with the FreeRADIUS server. There were multiple error messages when running Linux commands. We discovered this is normal — many of these errors don't actually interfere with FreeRADIUS functionality and can be safely ignored during initial setup.

// Conclusion

This lab showed how Fortinet's integrated wireless controller simplifies access point management. Deploying both WPA2 Personal and Enterprise on the same firewall — with a FreeRADIUS backend — demonstrates a realistic enterprise wireless setup. This skill is directly applicable in any environment that needs role-based or credential-based WiFi access control.

Lab #05

SSL VPN + Remote Desktop — Fortinet

FortiGateSSL VPNFortiClientFreeRADIUSRDP
// Purpose

The purpose of this lab is to configure an SSL/TLS VPN on a Fortinet firewall, adding an encrypted tunnel for remote network access and enabling remote desktop control from an external device.

// Background

Fortinet provides a security ecosystem of tightly integrated products — firewalls with built-in wireless LAN, AI-powered threat detection via FortiGuard, and cloud management tools. Their SSL VPN feature enables encrypted remote access using SSL/TLS, creating a secure tunnel that protects data in transit from hackers and malware.

Unlike traditional VPNs that require complex IPSec configurations, Fortinet's SSL VPN is managed entirely through the web GUI and uses FortiClient on the end device. This makes it accessible for both small offices and enterprise environments.

Key benefits include improved security, remote desktop access, safe data sharing, and lower total cost of ownership compared to traditional dedicated VPN appliances. Two-Factor Authentication (2FA) is supported but was not configured in this lab.

// Steps
  1. Go to User & Authentication → User Groups → Create New. Create a group named Remote Users, set type to Guest, and extend the expiration time beyond the default 4 hours.
  2. Go to User & Authentication → User Definition → Create New. Follow the multi-step wizard to create a VPN user. Note down the username and password — you will need these to connect.
  3. Go back to User & Authentication → User Groups → Remote Users. Add the new user as a member and define a RADIUS server for remote group authentication. Click OK.
  4. Boot up your FreeRADIUS server (from the previous lab).
  5. Go to VPN → SSL-VPN Settings. Set Remote Users under Tunnel Access (for full encryption). Direct all other users/groups to Web Access. Click Apply.
  6. Copy the SSL-VPN portal address shown in the settings (the blue highlighted URL).
  7. On your external device, navigate to the copied URL. Download and install FortiClient from the SSL-VPN portal.
  8. In FortiClient, press Configure VPN. Enter the portal address, give the VPN any name, enter your credentials, and click Connect.
  9. Verify the VPN connection: click the system tray arrow → right-click FortiClient → Open FortiClient Console. Confirm the VPN shows as connected.
  10. On the PC you want to remote into, enable Remote Desktop in Windows settings.
  11. On that PC, open Command Prompt and run ipconfig. Note the IP address.
  12. On your external device, open Remote Desktop Connection, enter the IP address, and click Connect. You should now have full remote desktop access through the VPN tunnel.
// Problems Encountered

The biggest issue was a misconfigured firewall policy. We could not establish remote desktop connectivity between the two PCs. After troubleshooting, we discovered we had set the SSL VPN traffic destination to the WAN interface instead of the LAN interface. Correcting this policy immediately resolved the issue.

// Conclusion

Configuring a Fortinet SSL VPN gave me a concrete understanding of how SSL/TLS encryption works in practice — not just in theory. Comparing this process to the Palo Alto VPN lab from earlier, I noticed Fortinet's workflow was more streamlined. The most valuable takeaway was the troubleshooting process: learning to read firewall policies carefully and trace exactly where traffic is being directed.
